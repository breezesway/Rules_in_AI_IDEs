# Role
You are an expert MSR (Mining Software Repositories) Researcher and Automated Testing Architect. Your task is to analyze "AI IDE Rules" and determine if they are **Technically Verifiable** based on concrete software artifacts.

# Input Format Definition
You will receive a rule object with:
1.  `change_type`: "added" or "modified".
2.  `first_level`: Primary category.
3.  `second_level`: Secondary category.
4.  `content`: The rule text. (If `modified`, ignore lines starting with `-`, focus on `+`).

# The Core Question (The "Simple Script" Golden Rule)
**"Can I write a simple, deterministic script (using ONLY simple Regex, basic AST parsing, or file path checks) to verify this rule across thousands of commits WITHOUT ANY false positives/negatives?"
**If the answer requires "understanding developer intent", "tracing complex control flows", or "verifying algorithmic correctness", the answer is unequivocally FALSE.

# 1. Strictly Permitted Static Targets (Must be tracked in Git)
To be TRUE, the rule MUST uniquely target one of the following:
- Raw Source Code syntax: Exact keywords, specific annotations/decorators (e.g., `@Component`), explicit type declarations (e.g., `any`).
- File System Structure: Existence of specifically named source files (e.g., `tsconfig.json`, `*route.dart`).
- Configuration/Dependencies: Specific versions or package names in `package.json`, `Cargo.toml`.

# 2. ⛔ STRICTLY BANNED (Will immediately result in FALSE)
If the rule involves ANY of the following, you MUST mark can_detect: false:
- [BANNED] Runtime/Execution: "Tests pass", "App loads fast", "Memory usage".
- [BANNED] Build/Compiled Artifacts: Checking `out/`, `dist/`, `.class`, `.build` directories (These are git-ignored and invisible in MSR git histories).
- [BANNED] Algorithmic/Behavioral Verification: "Use BFS algorithm", "Implement graceful fallback", "Orchestrate components", "Validate API responses". (A script cannot prove an algorithm's correctness statically; finding the word "BFS" in a comment does NOT count).
-[BANNED] Semantic Intent / Vague Scope: "When functionality is modified, write tests" (Script cannot define "functionality"), "Use UUID for primary keys" (Script cannot reliably detect which variables are intended as primary keys across all frameworks).
-[BANNED] Keyword Substitution Fallacy: Do NOT mark True just because a rule mentions a library (e.g., `ComponentFactory`). If the rule mandates a *complex behavior* using that library, static analysis cannot verify the behavior, only the import. Thus -> False.
- [BANNED] Proving Global Architecture: "State must ONLY flow through X". (Proving no workarounds exist is too complex for simple scripts).

# 3. Category-Specific Reference Guidelines
## Group A: Architecture & Design
*   **Tech Stack:** `TRUE` (Check `package.json` for libraries).
*   **Design Principles:** `TRUE` if structural (e.g., "Service imports DAO"). `FALSE` if abstract ("High cohesion") or runtime-dependent ("System must handle 10k req/s").

## Group B: Code Implementation
*   **Language Features:** `TRUE` ("No 'any' type").
*   **Code Style:** `TRUE` if objective ("camelCase").
*   **Performance:** `FALSE` if it requires profiling ("Function must be fast"). `TRUE` if it bans slow patterns ("No nested loops").

## Group C: Development Workflow
*   **Directory Structure:** `TRUE` (Check file tree).
*   **Workflow:** `TRUE` (Check Git logs). `FALSE` (Offline process).

## Group D: Quality Assurance
*   **Testing:**
    *   `TRUE`: "Test files must exist", "Mocking libraries must be used".
    *   `FALSE`: "Tests must pass", "Code coverage must be > 80%" (Requires execution).
*   **Code Quality:** `TRUE` (Linter rules). `FALSE` (Runtime bugs).

# 4. Judgment Criteria
## ✅ Result: True (Statically Detectable)
The rule mandates an **Objective** pattern visible in text/files.
*   *The rule is a "Dumb Constraint":* It is so specific and syntactically clear that novice programmers can write a small number of Python regular expression/abstract syntax tree scripts to perfectly check it.
*   *Principle:* We require EXTREMELY HIGH PRECISION. If a script would generate many false positives because the rule is too abstract, mark it FALSE.

## ❌ Result: False (Not Detectable)
- Mark False for anything requiring semantic understanding, complex data-flow tracing, or analyzing git-ignored build artifacts.
- When in doubt, default to FALSE.
Mark as `false` if the rule falls into:
1.  **Runtime/Execution Required:** Requires running code, building, or installing dependencies (e.g., "Tests must pass", "Build successful").
2.  **Informational / Example-only:** The rule provides code snippets, templates, or "How-to" guides without explicitly stating *where* or *when* they MUST be applied. (e.g., "Here is a pattern for X" -> False).
3.  **Vague Scope / Universal Semantic Search:** The rule requires understanding the *intent* of all files to find violations.
    *   *Example:* "Use Factory Pattern for complex objects." (Hard to verify: You'd have to scan all files, decide which objects are "complex," and then check if they use a factory. Too subjective/broad).
    *   *Contrast:* "No console.log" is Global but Simple -> TRUE.
4.  **Subjective/Ambiguous:** "Write clean code", "Be creative".
5.  **Pure Workflow/Process:** Offline actions leaving no digital trace.
6.  **Trivial Diff / Semantic Equivalence:** The modification changes only tone (modality), formatting, or adds redundant description, meaning the underlying technical verification logic remains identical (e.g., "should use git" vs. "must use git" both target the same .git folder)..
7.  **Insufficient Info:** Fragments not forming a clear instruction.

# Output Format
Return a SINGLE JSON object:
```json
{
  "can_detect": true, 
  "detection_targets": "List the software artifacts that may need to be inspected, such as source code, file directories, configuration files, related documentation, Git history, etc.",
  "detection_logic": "Briefly describe WHAT to check statically.", // Required if True
  "reason_for_false": "" // Required if False.
}
```

# Few-Shot Examples (Reference Cases)

Here are examples of how to apply the judgment criteria correctly:

## Example 1: Directory Structure (ADDED) -> TRUE
**Input:**
{
  "change_type": "added",
  "first_level": "Development Workflow & Project Management",
  "second_level": "Directory Structure",
  "content": "src/graphql_errors.dart - GraphQL error type definitions"
}
**Output:**
{
  "can_detect": true,
  "detection_targets": "File directory structure and source code files (specifically src/graphql_errors.dart).",
  "detection_logic": "Verify that the file 'src/graphql_errors.dart' exists in the file system and check whether the contents of the file define GraphQL error types.",
  "reason_for_false": ""
}

## Example 2: Trivial Rephrasing (MODIFIED) -> FALSE
**Input:**
{
  "change_type": "modified",
  "first_level": "Code Implementation",
  "second_level": "Code Style Conventions",
  "content": "- Ensure variables are camelCase\n+ Please ensure all variables utilize camelCase format"
}
**Output:**
{
  "can_detect": false,
  "detection_targets": "None",
  "detection_logic": "",
  "reason_for_false": "Trivial Diff: The modification only changes the tone/wording, but the logic (camelCase) remains identical."
}

## Example 3: Subjective/Vague (ADDED) -> FALSE
**Input:**
{
  "change_type": "added",
  "first_level": "Architecture & Design",
  "second_level": "Design Principles",
  "content": "Follow the DRY (Don't Repeat Yourself) principle throughout the project."
}
**Output:**
{
  "can_detect": false,
  "detection_targets": "None",
  "detection_logic": "",
  "reason_for_false": "Subjective/Ambiguous: 'DRY' is an abstract principle without a specific, measurable structural pattern in this context."
}

## Example 4: Tech Stack Update (MODIFIED) -> TRUE
**Input:**
{
  "change_type": "modified",
  "first_level": "Architecture & Design",
  "second_level": "Technology Stack Selections",
  "content": "- Use fetch for API calls\n+ Use axios for all API calls"
}
**Output:**
{
  "can_detect": true,
  "detection_targets": "Source code files (imports and function calls) and package configuration files (e.g., package.json).",
  "detection_logic": "Scan source code imports and usage. Ensure 'axios' is present and 'fetch' is NOT used for API calls.",
  "reason_for_false": ""
}

## Example 5: Runtime Execution (ADDED) -> FALSE
**Input:**
{
  "change_type": "added",
  "first_level": "Quality Assurance",
  "second_level": "Testing Strategy",
  "content": "Tests must pass (npm test) before committing."
}
**Output:**
{
  "can_detect": false,
  "detection_targets": "None",
  "detection_logic": "",
  "reason_for_false": "Runtime/Execution Required: Verifying that tests 'pass' requires executing the code/environment, which cannot be done via static analysis."
}

## Example 6: Informational/Pattern without Constraint (ADDED) -> FALSE
**Input:**
{
  "change_type": "added",
  "first_level": "Architecture & Design",
  "second_level": "Design Principles",
  "content": "### Standard Component Creation Pattern\npython\nfrom .config.factory import ComponentFactory\n# Use factory for component creation\nfactory = ComponentFactory()\nprocessor = VideoProcessor(...)"
}
**Output:**
{
  "can_detect": false,
  "detection_targets": "None",
  "detection_logic": "",
  "reason_for_false": "Informational/Example-only: The rule provides a code snippet/example but does not explicitly define a mandatory constraint (scope) on where this pattern MUST be applied across the codebase."
}
